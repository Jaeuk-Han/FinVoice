# CLAUDE.md — FinVoice (해외 금융뉴스 음성 브리핑 서비스)

이 파일은 이 저장소에서 작업하는 Claude(및 Sub Agent)를 위한 프로젝트 작업 안내서입니다.
코드 구조·과거 이력은 git과 `docs/`를 보면 되고, 이 파일은 **무엇을·어떤 규칙으로** 작업할지를 규정합니다.

---

## 1. 프로젝트 한 줄 설명

해외 영어 금융뉴스를 수집해 **Papago로 번역 → CLOVA Studio로 요약 → CLOVA Voice로 음성화**하여
"오늘의 증시 브리핑"을 웹으로 제공하는 NCP 기반 서비스. (수업 평가용 1주일 포트폴리오)

---

## 2. 대상 사용자와 핵심 기능

- **대상 사용자**:
  - **비로그인**: 누구나 접속 가능. 고정 관심종목(WATCHLIST) 브리핑 열람·검색만 가능.
  - **로그인**: 회원가입/로그인 후 개인 관심종목(최대 5개) 편집 가능. Finnhub에 존재하는 임의 종목 검색 허용.
- **핵심 기능**
  1. **정기 브리핑(배치)**: 매일 오전 7시 cron으로 고정 관심종목 뉴스를 자동 수집·번역·요약·음성화.
  2. **서비스 기동 시 자동 배치**: 서비스 시작 시 오늘 브리핑이 없으면 즉시 배치 실행(중복 방지).
  3. **온디맨드 검색**: 당일 캐시가 있으면 즉시 반환, 없으면 파이프라인 즉석 실행.
  4. **개인 관심종목 편집**: 로그인 사용자가 최대 5개 종목을 직접 설정. 저장 즉시 백그라운드에서 브리핑 자동 생성.
  5. **자동 새로고침**: 브리핑 생성 완료 시 프론트 폴링(`/api/today-item-count`)으로 자동 감지·새로고침.
  6. **실시간 주가 티커바**: 상단에 관심종목 실시간 주가(`/api/quotes`) 표시(60초 캐시).
- **고정 관심종목**: `app/config.py`의 `WATCHLIST`에만 둔다(현재 AAPL, TSLA, NVDA, MSFT, AMZN, GOOGL). 비로그인 기본값으로 사용.
- **신뢰성 원칙**: 요약에는 항상 출처 링크를 함께 저장·노출하고, 푸터에 "투자 조언 아님" 면책을 고정한다.

## 3. 기술 스택과 실행 명령

**스택**: Python 3.13 / FastAPI + uvicorn / Jinja2 (최소 UI) / PyMySQL / httpx / boto3(S3 호환) / pytest
(전체 핀 버전은 `requirements.txt` 참조)

**구조 요약**
- `pipeline/` — 공용 파이프라인(핵심): `fetch → translate → summarize → tts`, `runner.process_symbol`이 오케스트레이션.
- `app/` — FastAPI(`main.py`), 설정(`config.py`), DB(`db.py`), 인증(`auth.py`), 템플릿/정적파일.
- `batch_job.py` — cron 진입점, `pipeline.runner` 호출.
- `scripts/init_db.sql` — 전체 테이블 생성 DDL.
- `scripts/migrate_auth.py` — `user` · `user_watchlist` 테이블 추가 마이그레이션.

> **핵심 설계 포인트**: 번역·요약·TTS 로직은 전부 `pipeline/`에만 둔다. `batch_job.py`(배치), `/search`(온디맨드),
> 워치리스트 저장 후 백그라운드 스레드 — 세 진입점이 모두 같은 `runner`를 호출한다. 로직을 중복 작성하지 말 것.

**실행 명령** (PowerShell 기준; 서버는 동일 명령을 bash로 실행)
```powershell
# 의존성 설치
pip install -r requirements.txt

# 환경변수 준비 (.env 는 커밋 금지)
copy .env.example .env   # 값 채우기

# DB 초기화 (MySQL)
# mysql -h <host> -u <user> -p <db> < scripts/init_db.sql

# 웹앱 개발 실행
uvicorn app.main:app --reload --host 0.0.0.0 --port 8000

# 배치 1회 실행 (정기 브리핑 생성)
python batch_job.py

# 테스트 (외부 API는 전부 mock)
pytest -q
```

**완료 기준**: `pytest -q` green + 변경 영역 동작 증거 확보 후에만 완료를 보고한다.

## 4. NCP(Ncloud) AI/인프라 서비스와 역할

| 구분 | 서비스 | 역할 | 코드 위치 |
|------|--------|------|-----------|
| 번역 | **Papago** | 영어 기사(제목·본문 발췌) → 한국어 번역 | `pipeline/translate.py` |
| 요약 | **CLOVA Studio (HyperCLOVA X)** | 종목별 여러 기사를 묶어 핵심 요약 + 감성 라벨 산출 | `pipeline/summarize.py` |
| 음성 | **CLOVA Voice** | 한국어 요약문 → mp3 음성 생성 | `pipeline/tts.py` |
| 저장 | **Object Storage (S3 호환)** | 생성된 mp3 업로드 후 재생 URL 획득 (boto3) | `pipeline/tts.py` |
| DB | **Cloud DB for MySQL** | `briefing` / `briefing_item` / `article` / `user` / `user_watchlist` 저장·캐시 조회 | `app/db.py`, `scripts/init_db.sql` |
| 서버 | **Server (VM, Ubuntu)** | uvicorn 웹앱 + crontab 배치 실행 호스트 | 배포 대상 |
| 인증 | **API Gateway 키** | Papago/Voice 등 NCP API 공통 인증 헤더 | `.env`의 `NCP_APIGW_*` |
| 외부 | 영어 금융뉴스 API(Finnhub 등) | 원본 뉴스 수집(NCP 아님, 무료 티어 한도 주의) | `pipeline/fetch.py` |

**에러 처리 원칙**: 한 종목/단계 실패가 전체 배치를 죽이지 않게 한다(부분 성공 허용).
번역 실패 시 영어 원문으로라도 진행, **TTS 실패는 비치명적**(텍스트만 표시), 429(한도초과)는 재시도 없이 skip+로그.
같은 날·같은 종목은 재처리하지 않는다(캐시 우선).

## 5. Claude Design 참고 화면 목록과 디자인 톤

**화면 목록** (`app/templates/`, `app/static/style.css`)
1. **목록 페이지** (`/`, `list.html`) — 날짜별 브리핑 목록 + 상단 종목 검색 폼 + 실시간 주가 티커바.
2. **상세 페이지** (`/briefing/{id}`, `detail.html`) — 종목별 요약 카드(회사명·심볼·감성 배지·요약문·음성 플레이어·출처 토글).
3. **검색 결과** (`POST /search`) — 상세 페이지 템플릿 재사용(단일 종목 카드).
4. **로그인** (`/login`, `login.html`) — 이메일·비밀번호 입력, 오류 메시지 인라인 표시.
5. **회원가입** (`/register`, `register.html`) — 이메일·비밀번호 입력, 중복 이메일 오류 처리.
6. **관심종목 편집** (`/watchlist/edit`, `watchlist_edit.html`) — 칩(chip) UI로 최대 5개 종목 추가·삭제·저장.

**디자인 톤**
- **미니멀·정보 중심**: 군더더기 없는 단일 컬럼(`max-width: 760px`), system-ui 폰트. 화려한 장식보다 가독성 우선.
- **카드형 종목 블록**: 옅은 회색 보더 + 라운드(8px) 카드로 종목을 구분.
- **감성 색상 규칙**: positive `#00c471` / negative `#f04452` / neutral `#8c919e` (`style.css`의 `.s-*` 유지).
- **신뢰성 단서**: 출처 링크는 `<details>`로 접어두되 항상 노출, 푸터 면책 문구 고정.
- 디자인 변경 시 기존 클래스 규칙(`.s-positive` 등)과 한국어 UI 문구 톤을 유지하고, 외부 링크는 `rel="noopener noreferrer"`를 반드시 붙인다.

### Claude Design 작업 기준표

화면을 만들거나 수정할 때 아래 표를 기준으로 판단한다.

| 항목 | 기준 |
|------|------|
| **주요 사용자** | ① 비로그인: 빠르게 "오늘 시장/관심종목이 어땠는지"를 읽거나 듣고 싶어 함. 고정 종목 브리핑만 열람. ② 로그인: 관심종목을 직접 설정하고 개인화된 티커바·브리핑을 원함. |
| **첫 화면에서 바로 보여야 하는 것** | ① 가장 최근(오늘) 브리핑 진입점, ② 종목 심볼 검색 폼, ③ 날짜별 브리핑 목록. "투자 조언 아님" 면책은 항상 보이게. |
| **입력 화면에 필요한 필드** | 종목 심볼 1개(text, 예: `AAPL`). 대문자 자동 정규화, 지원 종목 안내(placeholder/힌트), 제출 버튼 1개. 그 외 필드 없음(YAGNI). |
| **결과 화면에 필요한 정보** | 날짜·종목(회사명+심볼), 감성 배지(positive/neutral/negative), 한국어 요약문, 음성 플레이어(있을 때만), 출처 링크(접이식), 면책 푸터. |
| **로딩 상태** | 온디맨드 검색은 파이프라인 즉석 실행으로 수 초 걸릴 수 있음 → 진행 중 표시(스피너/“요약 생성 중…”)와 중복 제출 방지(버튼 비활성). |
| **오류 상태** | 미지원 종목(400)·브리핑 없음(404)·파이프라인 실패를 사용자 친화 한국어 메시지로. 기술적 스택트레이스 노출 금지. |
| **빈 상태** | 브리핑이 아직 없을 때/검색 결과가 없을 때 빈 화면 대신 안내 문구 + 다음 행동(검색 유도) 제시. |
| **모바일에서 가장 중요한 버튼** | 검색(제출) 버튼과 음성 재생 버튼. 단일 컬럼에서 터치하기 쉬운 크기·간격, 화면 하단에서도 닿기 쉽게 배치. |
| **피해야 할 디자인** | ① 과한 장식(불필요한 그라데이션·애니메이션·아이콘 남발), ② 읽기 어려운 대비(옅은 회색 본문 텍스트, 배경과 비슷한 글자색 — 감성 색상도 충분한 명도 대비 유지), ③ 실제처럼 보이는 더미 개인정보(실명·이메일·전화번호·실제 계좌/금융 데이터). 예시는 명백한 가짜 값만 사용(§8 참조). |

## 6. Sub Agent 호출 규칙

- **직접 처리**: 단일 파일 소규모 수정, 명령 1개, 간단한 질문.
- **위임(Sub Agent)**: 다중 파일 변경·리팩터·디버깅·리뷰·계획·리서치·검증.
- **라우팅**
  - 구현 작업 → `executor` (복잡하면 `model=opus`).
  - 코드 리뷰 → `code-reviewer`, 보안 점검 → `security-reviewer`.
  - 완료 검증 → `verifier` (작성과 검증은 반드시 다른 패스로 분리, 자기 승인 금지).
  - NCP/외부 SDK·API 사용법이 불확실하면 → `document-specialist`(저장소 문서 우선, 그다음 웹).
- **병렬화**: 독립적인 2개 이상 작업은 한 번에 병렬 호출. 빌드/테스트는 `run_in_background`.
- **계획 우선**: 새 기능·다단계 작업은 먼저 계획(plan)을 세운 뒤 실행한다.

## 7. 서버 반영 규칙

- SSH 또는 배포 관련 요청은 반드시 계획부터 시작한다.
- 서버 주소, 계정명, 키 경로, 비밀번호, API 키는 CLAUDE.md에 직접 쓰지 않는다.
- 먼저 `SERVER_RUNBOOK.md`에 실행 위치, 포트, 환경변수, 실행 명령, 롤백 방법을 정리한다.
- 실제 반영 전에는 로컬 실행, 비밀값 노출 검사, 발표 데모 흐름을 확인한다.
- 서버에서 명령을 실행하기 전에는 어떤 명령을 왜 실행하는지 사용자에게 설명한다.

## 8. 금지사항 (하드코딩 절대 금지)

다음 값은 **소스 코드·문서·커밋·테스트 어디에도** 하드코딩하지 않는다. 모두 `.env`(git 제외) 또는 안전한 비밀 저장소로만 다룬다.

- ❌ **API 키 / 토큰**: 뉴스 API 키, NCP API Gateway 키, CLOVA Studio/Voice/Papago 키, Object Storage Access/Secret 키.
- ❌ **DB 자격증명**: 실제 호스트·사용자·비밀번호.
- ❌ **SSH 개인키 / 서버 비밀번호 / 접속 정보**.
- ❌ **실제 개인정보**(PII): 실명, 이메일, 전화번호 등.
- ❌ **실제 금융 데이터**: 실제 계좌·체결·포지션 등 민감 금융 데이터.

**대신**:
- 비밀값은 `os.getenv` / `config.get_env`로 로딩하고, `.env.example`에는 자리표시자(`your_xxx_key`)만 둔다.
- 테스트는 외부 API를 **mock**하며 가짜/예시 값만 사용한다.
- 새 비밀 항목이 필요하면 `.env.example`에 키 이름만 추가하고 발급 방법은 README/문서에 적는다.
- 실수로 비밀값이 커밋에 들어가면 즉시 알리고 회전(rotate)을 권고한다.

## 9. 요청 말투별 작업 규칙

- "가볍게 봐줘", "오탈자만 봐줘"라고 말하면 메인 Claude가 혼자 확인한다.
- "반영해줘", "한 화면 수정해줘"라고 말하면 파일을 먼저 읽고 작은 수정 계획을 세운 뒤 진행한다.
- "회의해줘", "토론해줘", "깊게 검토해줘"라고 말하면 필요한 역할만 불러 의견을 모은다.
- "서버", "배포", "보안", "비밀값"이 들어간 요청은 바로 수정하지 않고 먼저 점검표와 계획을 만든다.

외부 확인이 필요한 내용은 `/mcp`와 `/skills`를 먼저 확인한다.
확인 도구가 없으면 추정하지 말고 "확인이 필요함"을 남긴다.
모든 역할을 자동으로 부르지 말고, 요청에 맞는 2~5개 역할만 고른다.
